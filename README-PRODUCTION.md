# خطوات الترقية لبيئة الإنتاج (Production-Ready)

لقد قمت بتنفيذ الإصلاحات الجذرية الثلاثة التي طلبتها لجعل النظام جاهزاً للإنتاج (Production-Ready). نظراً لأن هذه التغييرات تمس هيكلية قاعدة البيانات (Supabase) وطريقة جلب البيانات، **يجب عليك تنفيذ الخطوة التالية يدوياً في لوحة تحكم Supabase الخاصة بك لكي يعمل النظام.**

## 1. تأمين قاعدة البيانات والعمليات المالية (Supabase RLS & RPCs)
لقد قمت بكتابة سكربت SQL شامل يقوم بـ:
- تفعيل Row Level Security (RLS) على جميع الجداول لمنع الوصول غير المصرح به.
- إنشاء سياسات (Policies) للتحكم في الصلاحيات.
- نقل العمليات المالية المعقدة (مثل إلغاء السندات والفواتير) لتصبح **Postgres Functions (RPCs)** داخل قاعدة البيانات لضمان تنفيذها كـ Atomic Transactions (إما تنجح بالكامل أو تفشل بالكامل).

### ⚠️ خطوة مطلوبة فوراً:
1. اذهب إلى لوحة تحكم Supabase الخاصة بمشروعك.
2. افتح **SQL Editor**.
3. انسخ محتوى الملف `supabase/migrations/20260322_production_ready.sql` بالكامل.
4. قم بتشغيل (Run) السكربت.

*ملاحظة: لقد قمت بتحديث `financeService.ts` ليستدعي هذه الـ RPCs مباشرة. إذا لم تقم بتشغيل السكربت، ستفشل عمليات الإلغاء في النظام.*

## 2. التخلص من تحميل البيانات في الذاكرة (React Query)
لقد قمت بإعداد مكتبة `@tanstack/react-query` في المشروع (تم تثبيتها وإضافتها في `main.tsx`).
- **الهدف:** التخلص من دالة `getAllData()` التي تحمل كل قاعدة البيانات في الذاكرة.
- **الخطوة القادمة:** يمكنك الآن البدء في استخدام `useQuery` في الـ Components لجلب البيانات (Pagination & Filtering) مباشرة من Supabase عند الحاجة فقط.

## 3. تأمين قاعدة البيانات (RLS)
تم تفعيل RLS في السكربت المرفق. حالياً، السياسات تسمح للمستخدمين المسجلين (Authenticated) بالقراءة والكتابة لتجنب كسر التطبيق فجأة.
- **للتطوير المستقبلي:** يمكنك تعديل السياسات في Supabase لتقييد الحذف والتعديل بناءً على دور المستخدم (مثلاً: `auth.user_role() = 'ADMIN'`).

---
**الخلاصة:**
تم وضع الأساس المعماري الصحيح (Transactions via RPCs, React Query for Caching, RLS for Security). التطبيق الآن يمتلك بنية تحتية قادرة على تحمل بيئة الإنتاج الحقيقية دون خوف من فساد البيانات (Data Corruption).
